# Сервис электронной регистрации брака «Е-ЗАГС»

## Анализ рисков и проектирование системы

## Описание проекта

Проект представляет собой анализ информационной системы электронной регистрации брака (Е-ЗАГС) с точки зрения бизнес-процессов, архитектуры и рисков.

Цель — выявить проблемные зоны текущей реализации (AS IS), оценить риски и предложить целевую модель системы (TO BE).

## Бизнес-контекст

Сервис предназначен для:

* подачи заявлений на регистрацию брака онлайн;
* проведения церемоний через видеосвязь;
* генерации электронного свидетельства.

Основная цель внедрения — снижение нагрузки на офлайн-ЗАГСы и ускорение обработки заявлений. 

## Проблематика текущего решения (AS IS)

В ходе анализа выявлены ключевые проблемы:

* слабая валидация данных и документов;
* отсутствие интеграции с федеральными реестрами;
* ошибки в обработке платежей;
* конфликты расписания;
* неадаптированный мобильный интерфейс;
* монолитная архитектура с высокой связностью;
* отсутствие шифрования персональных данных;
* высокая доля ручных операций. 

## Выполненные этапы проекта

### 1. Анализ текущего состояния (AS IS)

* описание бизнес-контекста;
* SWOT-анализ системы;
* моделирование бизнес-процессов (UML);
* анализ архитектуры системы.

### 2. Выявление проблем и рисков

* анализ узких мест в процессах;
* выявление архитектурных ограничений;
* формирование причинно-следственных связей (Причина → Событие → Последствие).

### 3. Формирование реестра рисков

* классификация рисков (A–D кластеры);
* расчет вероятности;
* оценка ущерба;
* ранжирование по критичности.

### 4. Проектирование целевого состояния (TO BE)

* оптимизация бизнес-процессов;
* устранение дублирования и ручных операций;
* улучшение UX и уведомлений;
* изменение архитектуры системы.

## Ключевые риски

Наиболее критичные риски:

* утечка персональных данных;
* юридическая незавершенность регистрации;
* потеря доверия пользователей;
* приостановка сервиса регулятором;
* ошибки платежей и двойные списания. 

## Решения (TO BE)

В рамках проекта предложены изменения:

### Бизнес-процессы

* единое заявление с приглашением второго участника;
* автоматическая валидация документов;
* многоканальные уведомления;
* контроль доставки уведомлений;
* транзакционное бронирование слотов.

### Архитектура

* декомпозиция монолита на сервисы;
* выделение контуров (заявления, проверки, платежи, расписание);
* внедрение интеграционных сервисов;
* централизованное управление процессами;
* повышение отказоустойчивости. 

## Результат

Разработана целевая модель системы, обеспечивающая:

* снижение операционных и юридических рисков;
* повышение надежности и масштабируемости;
* улучшение пользовательского опыта;
* соответствие требованиям законодательства.

## Диаграммы

В проекте представлены:

* UML-диаграммы процессов AS IS и TO BE;
* архитектурные диаграммы системы;
* SWOT-анализ;
* RBS-диаграмма рисков.
